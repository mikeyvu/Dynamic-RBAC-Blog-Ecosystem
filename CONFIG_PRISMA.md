# Prisma 7 + NestJS + PostgreSQL in Docker — Full Setup Guide

## Architecture Overview

Prisma 7 is fundamentally different from Prisma 5/6. It has 3 separate layers:

```
CLI commands (migrate, generate)  → reads from prisma.config.ts
Runtime (NestJS app queries)      → uses driver adapter (PrismaPg)
schema.prisma                     → defines models only, NO url field
PrismaClient class                → generated to src/generated/prisma/, NOT from @prisma/client
```

---

## Step 1 — Docker Compose for PostgreSQL

**`docker-compose.yml`** at project root:
```yaml
version: '3.8'
services:
  postgres:
    image: postgres:15-alpine
    environment:
      POSTGRES_USER: ${POSTGRES_USER}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
      POSTGRES_DB: ${POSTGRES_DB}
    ports:
      - "5433:5432"   # host:container — use 5433 to avoid conflicts with local postgres
    volumes:
      - postgres_data:/var/lib/postgresql/data
volumes:
  postgres_data:
```

**`.env`** at project root (for Docker Compose only):
```
POSTGRES_USER=your_user
POSTGRES_PASSWORD=your_password
POSTGRES_DB=your_db_name
```

Start the container:
```bash
docker compose up -d
```

## Step 2 — Install Dependencies (inside /backend)

```bash
npm install prisma @prisma/client @prisma/adapter-pg --save-dev
npm install @prisma/client @prisma/adapter-pg
```

- `prisma` = CLI tool (devDependency)
- `@prisma/client` = runtime library
- `@prisma/adapter-pg` = PostgreSQL driver adapter (required in Prisma 7)

---

## Step 3 — Initialize Prisma

```bash
npx prisma init
```

This creates: `prisma/schema.prisma`, `backend/.env`, and `prisma.config.ts`.

---

## Step 4 — Configure `backend/.env`

```
DATABASE_URL="postgresql://your_user:your_password@localhost:5433/your_db_name?schema=public"
```

Port `5433` must match the host port in `docker-compose.yml`.

## Step 5 — Configure `prisma/schema.prisma`

```prisma
generator client {
  provider     = "prisma-client"          # Prisma 7 new generator
  output       = "../src/generated/prisma"
  moduleFormat = "cjs"
}

datasource db {
  provider = "postgresql"
  # NO url field — Prisma 7 removed url from schema.prisma
}

# Define your models below...
```

## Step 6 — Verify `prisma.config.ts`

Auto-generated by Prisma, but confirm it looks like this:

```typescript
import 'dotenv/config';
import { defineConfig } from '@prisma/config';

export default defineConfig({
  schema: 'prisma/schema.prisma',
  migrations: { path: 'prisma/migrations' },
  datasource: { url: process.env['DATABASE_URL'] },  // CLI reads URL from here
});
```

## Step 8 — Create `src/prisma/prisma.module.ts`

```typescript
import { Global, Module } from '@nestjs/common';
import { PrismaService } from './prisma.service';

@Global()
@Module({
  providers: [PrismaService],
  exports: [PrismaService],
})
export class PrismaModule {}
```

`@Global()` makes PrismaService injectable anywhere without re-importing the module.

---

## Step 9 — Create `src/prisma/prisma.service.ts`

Three critical rules for Prisma 7:

1. Import `PrismaClient` from `'../generated/prisma/client'`, NOT from `'@prisma/client'`
2. Use adapter pattern: `new PrismaPg({ connectionString: url })`, NOT `datasourceUrl`
3. Use **composition** (not `extends`) — Prisma 7 exports `PrismaClient` as a `const`, which breaks TypeScript type inference when used with `class extends`

```typescript
import { Injectable, OnModuleInit, OnModuleDestroy } from '@nestjs/common';
import { PrismaClient } from '../generated/prisma/client';
import { PrismaPg } from '@prisma/adapter-pg';

@Injectable()
export class PrismaService implements OnModuleInit, OnModuleDestroy {
  private readonly client: PrismaClient;

  // Add a getter for each model defined in schema.prisma
  get user() { return this.client.user; }
  get role() { return this.client.role; }
  get post() { return this.client.post; }
  // get $transaction() { return this.client.$transaction.bind(this.client); }

  constructor() {
    const url = process.env.DATABASE_URL;
    if (!url || url.trim() === '') {
      throw new Error('DATABASE_URL is not configured in .env!');
    }
    this.client = new PrismaClient({
      adapter: new PrismaPg({ connectionString: url }),
    });
  }

  async onModuleInit() { await this.client.$connect(); }
  async onModuleDestroy() { await this.client.$disconnect(); }
}
```

## Step 11 — Fix ESLint False Positives

Prisma 7 generated files use `// @ts-nocheck`, which causes `@typescript-eslint/no-unsafe-*` rules to fire on `prisma.service.ts` as false positives. Add this override in `eslint.config.mjs`:

```js
{
  files: ['src/prisma/prisma.service.ts'],
  rules: {
    '@typescript-eslint/no-unsafe-member-access': 'off',
    '@typescript-eslint/no-unsafe-return': 'off',
    '@typescript-eslint/no-unsafe-call': 'off',
    '@typescript-eslint/no-unsafe-assignment': 'off',
  },
},
```

## Common Mistakes in Prisma 7

| Wrong | Correct |
|-------|---------|
| `import { PrismaClient } from '@prisma/client'` | `import { PrismaClient } from '../generated/prisma/client'` |
| `super({ datasourceUrl: url })` | `new PrismaClient({ adapter: new PrismaPg({ connectionString: url }) })` |
| `url = env("DATABASE_URL")` in schema.prisma | No `url` in datasource block |
| `class PrismaService extends PrismaClient` | Composition with `private readonly client: PrismaClient` |
